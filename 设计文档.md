## 一、项目概述

初中编程竞赛项目，构建一个地铁运营数据分析系统。基于杭州地铁1号线（简化版14站）的模拟运行数据，实现多维可视化分析、异常检测和优化建议自动生成。

**技术栈**: Python (Pandas + Matplotlib/Seaborn + Tkinter + scikit-learn)
**实现路线**: 采用 Python 单机 GUI 方案，不采用 JavaScript/Node.js 路线
**MVP 范围**: F1.1-F4.4 为第一阶段必做；API、预测、交互增强、导出为第二阶段增强项

**评分维度**: 数据处理(35%) | 可视化(30%) | 分析深度(25%) | 工程实现(10%)

### 3.2 数据生成算法

**客流模型 — 双峰高斯分布**:

```python
def base_passenger_curve(hour: int, day_type: str) -> float:
    """返回 0-1 的归一化客流系数"""
    if day_type == "workday":
        # 早高峰 8:00 峰值，晚高峰 18:00 峰值
        morning = math.exp(-0.5 * ((hour - 8.0) / 0.8) ** 2)
        evening = math.exp(-0.5 * ((hour - 18.0) / 1.0) ** 2)
        base = max(morning, evening)
        # 午间小高峰
        noon = 0.3 * math.exp(-0.5 * ((hour - 12.5) / 1.2) ** 2)
        return max(base, noon, 0.08)  # 最低 8% 基底客流
    else:
        # 周末：10-14 点和 16-20 点平缓双峰
        mid_day = 0.6 * math.exp(-0.5 * ((hour - 12.0) / 2.5) ** 2)
        afternoon = 0.5 * math.exp(-0.5 * ((hour - 17.0) / 2.0) ** 2)
        return max(mid_day, afternoon, 0.1)
```

**延误模型**:

```python
def generate_delay(passengers: int, hour: int) -> float:
    """延误 = 基底 + 客流相关项 + 随机噪声"""
    base_delay = 0.5
    crowding_factor = (passengers / 5000) * 2.0
    peak_factor = 1.5 if hour in [7, 8, 17, 18] else 0.5
    noise = random.gauss(0, 0.3)
    return max(0, base_delay + crowding_factor * peak_factor + noise)
```

**车速模型**:

```python
def generate_speed(crowding: float) -> float:
    """正常 38-42 km/h，拥挤时降至 28-32 km/h"""
    base_speed = 40.0
    speed_reduction = crowding * 12.0
    noise = random.gauss(0, 1.5)
    return max(20, base_speed - speed_reduction + noise)
```

**拥挤度**: `crowding = min(1.0, passengers / capacity)`，大站容量5500，普通站4000

### 4.3 图表嵌入（v2 修正：release-driven 滑块）

**[v2 修正]** 不使用 `ttk.Scale(command=...)`（会在拖动中持续触发），改用 `<ButtonRelease-1>` 事件绑定：

```python
from matplotlib.backends.backend_tkagg import FigureCanvasTkAgg
from matplotlib.figure import Figure

class DashboardFrame(ttk.Frame):
    def __init__(self, parent):
        super().__init__(parent)
        self.fig = Figure(figsize=(8, 5), dpi=100)
        self.canvas = FigureCanvasTkAgg(self.fig, master=self)
        self.canvas.get_tk_widget().pack(fill=tk.BOTH, expand=True)
        
        # 时间滑块 — release-driven
        self._slider_var = tk.IntVar(value=8)
        self.time_slider = ttk.Scale(
            self, from_=6, to=23, orient=tk.HORIZONTAL,
            variable=self._slider_var,
            command=self._on_slider_drag,   # 拖动中仅更新标签
        )
        self.time_slider.pack(fill=tk.X, padx=10, pady=5)
        # 松开鼠标时才触发图表重绘
        self.time_slider.bind("<ButtonRelease-1>", self._on_slider_release)
        
        # 当前时段标签
        self.hour_label = ttk.Label(self, text="当前时段: 8:00")
        self.hour_label.pack()
    
    def _on_slider_drag(self, value):
        """拖动过程中仅更新标签文字，不触发重绘"""
        hour = int(float(value))
        self.hour_label.config(text=f"当前时段: {hour}:00")
    
    def _on_slider_release(self, event):
        """松开鼠标后触发图表重绘"""
        hour = self._slider_var.get()
        self._refresh_charts_for_hour(hour)
```

**[v3 修正]** API 数据需进入分析链路中的“校验支路”，不能仅作为装饰性接口，但也不直接改写 F3 核心排行。

| API | 用途 | 端点 |
|-----|------|------|
| 公交路径规划 | **获取站间实际行驶时长**，转换为平均车速 | `GET /v3/direction/transit/integrated` |
| 关键词搜索 | 获取站点经纬度（用于路径规划的起终点） | `GET /v3/place/text` |

**数据进入分析链路的方式**:
1. `amap_client.fetch_line_speeds()` 获取所有相邻站区间在采样小时的实际行驶时长
2. `data_loader.build_api_validation_table()` 将区间车速汇总为 `hour × api_avg_speed_kmh`
3. `analyzer.compare_sim_vs_api()` 对比 `avg_speed_kmh`（模拟小时均速） vs `api_avg_speed_kmh`（实际小时均速）
4. 偏差超过 `20%` 的采样小时标记为"模拟偏差警告"
5. `report_generator.generate_api_comparison()` 在报告中增加"模拟 vs 实际对比"段落

**集成策略**:
- 默认模式：无 API key，使用内置数据，所有功能正常
- 增强模式：有 key 时从 API 获取实际数据，进入分析链路
- 错误处理：API 失败时 graceful fallback，状态栏提示用户

## 六点五、关键口径与交互约定

### 6.5.1 最低效时段算法

1. 对当前筛选后的数据按 `hour` 聚合，得到每小时平均延误和平均车速。
2. 计算延误归一化：
   `delay_norm = (delay - delay_min) / (delay_max - delay_min)`。
3. 计算车速损失：
   `speed_loss = 40 - avg_speed_kmh`，其中 `40km/h` 为基准速度。
4. 计算车速损失归一化：
   `loss_norm = (speed_loss - loss_min) / (loss_max - loss_min)`。
5. 计算效率得分：
   `score = 0.6 * delay_norm + 0.4 * loss_norm`。
6. 得分最高的小时即"最低效时段"；若分母为 0，则对应归一化项记为 0。

### 6.5.2 瓶颈项命名

- 当前数据粒度为"站点 × 小时"，不含相邻站区间数据。
- 因此分析结果统一命名为"瓶颈站点时段 Top 3"，不使用"瓶颈路段"表述。

### 6.5.3 时间滑块交互

- 滑块范围为 `6-23`。
- 拖动过程中只更新标签，不触发整图重绘。
- 用户松开滑块后才刷新图表标记和报告摘要。
- 热力图高亮对应小时列；折线图、柱状图增加该小时的参考标记，不改动底层统计口径。
- **实现方式**：不使用 `ttk.Scale(command=...)`，而是绑定 `<ButtonRelease-1>` 事件。

### 6.5.3.1 [v3 新增] 默认参数

- 固定随机种子：`20260407`
- 默认日期筛选：`全部`
- 默认站点筛选：`全部站点`
- 默认主图表：`热力图`
- 默认当前关注时段：`8:00`

### 6.5.4 建议生成规则

- 第一条建议基于最拥堵站点，输出客流疏导或增开运力建议。
- 第二条建议基于最低效时段，输出调度优化建议。
- 第三条建议基于准点率或瓶颈站点时段，输出运营改善建议。
- 每条建议必须包含至少一个具体指标，例如站名、时段、准点率、平均延误或平均车速。
