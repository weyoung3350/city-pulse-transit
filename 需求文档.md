## 二、项目背景

城市交通拥堵是常见的社会问题。本项目以杭州地铁1号线（简化版14站）为分析对象，模拟并分析地铁线路的运行数据，找出低效环节，自动生成可视化分析报告和优化建议，帮助理解公共交通运营效率问题。

### 5.2 数据生成规则

| 规则 | 描述 |
|------|------|
| 时间范围 | 7天（5工作日 + 2周末），每天 06:00-23:00（18个时段） |
| 客流规律 | 工作日双峰高斯（早8:00 σ=0.8，晚18:00 σ=1.0），周末平缓双峰 |
| 延误规律 | 高峰 + 大站 → 延误概率高，基底0.5分钟 |
| 车速规律 | 基准40km/h，拥挤时降至28-32km/h |
| 拥挤度 | 客流/容量（大站5500，普通站4000），上限1.0 |
| 准点标准 | 延误 < 2分钟视为准点 |
| 随机性控制 | 使用固定随机种子生成模拟数据，保证演示和复现结果一致 |

### 5.3 关键计算口径

| 项目 | 定义 |
|------|------|
| 最低效时段聚合粒度 | 对当前筛选结果按 `hour` 聚合，计算每小时平均延误与平均车速 |
| 延误归一化 | `(当前小时平均延误 - 最小平均延误) / (最大平均延误 - 最小平均延误)`；若分母为 0，则记为 0 |
| 车速损失归一化 | `((40 - 当前小时平均车速) - 最小损失) / (最大损失 - 最小损失)`；其中 40km/h 为基准速度；若分母为 0，则记为 0 |
| 效率得分 | `0.6 × 延误归一化 + 0.4 × 车速损失归一化`，得分最高者为最低效时段 |
| 最拥堵站点 | 对当前筛选结果按站点聚合，比较平均拥挤度；并同时输出峰值拥挤度与总客流 |
| 瓶颈站点时段 | 对当前筛选结果按 `station + hour` 聚合，比较平均车速，取最低的前 3 项 |
| 优化建议生成 | 按"拥堵疏导、低效调度、准点率改善"三类模板生成，每条建议必须包含至少 1 个具体指标 |

### 5.4 [v3 新增] API 校验数据粒度与映射规则

| 项目 | 定义 |
|------|------|
| API 原始粒度 | `segment × sample_hour`，即相邻站区间在某采样小时的实际行驶时长/距离 |
| 校验汇总粒度 | `hour`，将同一采样小时的所有区间按距离加权汇总为整线实际均速 |
| 与模拟数据的对齐方式 | 将模拟数据按 `hour` 聚合出整线平均车速，再与 API 的 `hour` 级实际均速比较 |
| 纳入报告的位置 | “模拟 vs 实际”校验段落 + 偏差警告列表 |
| 不纳入的分析项 | F3.1 最拥堵站点、F3.2 最低效时段、F3.3 瓶颈站点时段的核心排行仍基于模拟数据 |

### 5.5 [v3 新增] 当前关注时段口径

| 项目 | 定义 |
|------|------|
| 当前关注时段默认值 | `8:00` |
| 报告区摘要字段 | 当前时段、当前筛选范围内总客流、平均延误、平均车速、该时段最拥堵站点 |

| 折线图行为 | 仅增加当前小时参考线，不改曲线统计口径 |
| 柱状图行为 | 保持“每日准点率”数据不变，仅在标题或角标显示“当前关注时段：X:00”，不按小时重算柱体 |

## 七、外部接口

### 7.1 高德地图 Web 服务 API（可选）

**[v2 修正]** API 需获取可进入分析链路的实际交通数据，而非仅补充静态元数据。

| 接口 | 用途 | 请求方式 |
|------|------|----------|
| 公交路径规划 `/v3/direction/transit/integrated` | 获取站间实际行驶时长和换乘等待时间，转换为实际车速与模拟数据对比校验 | GET |
| 关键词搜索 `/v3/place/text` | 查询地铁站经纬度（用于路径规划的起终点参数） | GET |

**数据进入分析链路的方式**：
1. 将 API 返回的相邻站区间行驶时长转换为区间平均车速
2. 按 `hour` 聚合为整线实际均速校验表
3. 在报告中增加"模拟 vs 实际"对比段落，展示数据校验结果
4. 若 API 数据与模拟数据偏差超过 `20%`，在报告中标记“模拟偏差警告”

**约束**：
- 需用户自行申请 API Key 并填入 `config.py`
- 调用频率限制：每秒不超过5次
- 无 Key 时全部功能正常运行（使用内置数据）
- API 数据仅作为校验/补充，不替代模拟数据的主分析流程
- API 校验默认不参与 F3 核心异常排行，只参与对比校验与报告提示
